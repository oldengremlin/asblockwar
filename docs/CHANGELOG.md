## [3.9.3] — 2026-07-23

### Додано

- **GUI — «Dry Run + Graph»**: третій пункт у меню `SplitMenuButton`. Запускає всю
  обробку в режимі dry-run (жодних файлів не записується, `AfterCommand` не виконується),

  `isDependencyGraph() && (!isDryRun() || isDryRunWithGraph())`.
  - Заголовок діалогу: «ASBlockWar — Dry Run + Graph»
  - Статус: «Running (dry run + graph)...» → «Dry run + graph complete.»
  - Після завершення кнопка «Dependency» активується (файл графа щойно оновлено).

## [3.9.2] — 2026-07-23

### Додано

- **GUI — Dry Run через `SplitMenuButton`**: кнопка «Run» замінена на `SplitMenuButton`.
  Лівий кліком — звичайний Run (без змін поведінки). Стрілка праворуч розкриває меню
  з пунктом **«Dry Run»** — запускає повну обробку з тимчасовим прапором `dryRun=true`,

  - Граф залежностей у режимі Dry Run **не генерується** (поведінка ядра незмінна).
  - Кнопка «Dependency» після Dry Run активується лише якщо файл графа вже існує
    (від попереднього звичайного Run) — раніше перевірявся лише налаштований шлях.

## [3.9.1] — 2026-07-22

### Виправлено

- **Бекап `list.txt` — хибне пропускання при відсутніх бекапах**: у 3.9.0 порівняння
  виконувалося з поточним `list.txt`, а не з останнім бекапом. Через це, якщо всі бекапи
  були видалені, новий бекап не створювався навіть при першому `Run`. Тепер
  `storeAggressorAsnResources()` викликає `findLatestBackup()`, що шукає найновіший файл
  з іменем виду `list.TIMESTAMP.txt` у директорії бекапів; якщо жодного бекапу немає —
  копія завжди створюється. Замість `Files.move` використовується `Files.copy`, що
  коректно працює між різними файловими системами.

- **`ForceASBlock` — список не сортувався при збереженні**: у `Config.save()` список
  `ForceASBlock` тепер сортується чисельно за номером ASN (аналогічно до `PrimaryEnemyResources`
  для AUT-NUM). AS-SET-ів у `ForceASBlock` не буває — тому застосовується лише числове
  сортування.

### Додано

- **Властивість `ListFileBackupDir`**: нова конфігураційна властивість задає директорію,
  куди зберігаються резервні копії `list.txt`. За замовчуванням — порожній рядок (бекапи
  зберігаються поруч з `list.txt`). Доступна через:
  - CLI: `--list-file-backup-dir <path>`
  - Properties: `ListFileBackupDir=<path>`
  - GUI: нове поле «List backup directory:» з кнопкою вибору директорії у діалозі
    Properties. Корисно для збереження бекапів на окремому розділі або у мережевому
    сховищі.

## [3.9.0] — 2026-07-22

### Покращено

- **Properties — сортування `PrimaryEnemyResources` при збереженні**: при кожному `Save`
  перелік автоматично впорядковується: спочатку AS-SET-и (не `AS\d+`) за алфавітом,
  потім AUT-NUM (`AS\d+`) за числовим номером. При введенні нових записів порядок не
  змінюється — лише при записі у файл конфігурації.

- **Properties — перевірка дублікатів при додаванні**: кнопка `+` у списках
  **PrimaryEnemyResources**, **BlockCountry**, **ForceASBlock** та **ForceNETBlock**
  тепер мовчки ігнорує спроби додати елемент, що вже є у списку (порівняння після
  нормалізації регістру та формату). Один рядок у `promptAdd()` покриває всі чотири
  списки.

- **Бекап `list.txt` — пропуск при незмінному вмісті**: `storeAggressorAsnResources()`
  тепер порівнює новий вміст з поточним `list.txt` перед створенням резервної копії.
  Якщо вміст ідентичний (повторний `Run` без змін у списку), бекап і перезапис
  пропускаються з відповідним повідомленням у лозі. Порівняння рядкове — без SHA,
  достатньо для файлу такого розміру; відбувається під захистом FileLock, тому
  гонок немає.

  при натисканні Reset. Працює однаково в sfdp- і D3 force-simulation-режимах.

## [3.8.0] — 2026-07-20

### Додано

- **Reporter — деталі змін**: після компактної таблиці «Вилучено / Додано / Змінено» тепер виводиться
  розширена таблиця з рядком на кожен змінений ASN — колонки: ASN, Дія, CN (country), Організація
  (org-name з RPSL). Ширина колонок адаптивна. Дозволяє одразу бачити мотивацію кожної зміни,
  не звертаючись до файлів STORE/.

- **Граф — фільтр за типом вузла**: у бічній панелі фільтрів додано групу чекбоксів
  **ASN / AS-SET / MNTNER / ORG**. Працює незалежно від фільтра за статусом — вузли прихованого
  типу зникають разом зі всіма ребрами, що їх торкаються. Кнопка Reset відновлює обидва фільтри.

- **Граф — пошук по полю details**: поле пошуку вгорі тепер збігається не лише з ідентифікатором
  вузла, а й з його полем `details` (org-name, country, as-name тощо). Зручно для пошуку по
  імені організації без знання точного ASN.

- **Граф — кнопка «Копіювати шлях» у плаваючому вікні**: у заголовку кожного плаваючого вікна
  маршруту з'явилася кнопка ⎘. По кліку шлях копіюється у буфер обміну у вигляді
  `AS12345 → AS-MAILRU → AS60781`. Кнопка на 1,5 с показує ✓ як підтвердження. Використовує
  `navigator.clipboard` з fallback на `execCommand('copy')` для старих браузерів.

- **`--dry-run` / `-n` — режим симуляції**: новий прапорець командного рядка (та GUI). При
  активації вся обробка виконується повністю (запити до БД, фільтрація, виявлення AS, звіт),
  але жоден файл на диску не записується і `AfterCommand`-скрипт не запускається. Зручно для
  перевірки того, що знайде інструмент, без ризику зміни робочого оточення. Прапорець
  транзитивний — автоматично враховується у `FileUtils.writeStoreFile()`, у backup-кроці
  `storeAggressorAsnResources()`, у `BatchRunner` та у генерації графа залежностей.

## [3.7.0] — 2026-07-20

### Додано

- **Властивість `PrimaryEnemyResources`**: захардкоджений список AS-SET (`AS-MAILRU`,
  `AS-VKONTAKTE`, `AS-VK`, `AS-YANDEX`, `AS-M100`) перенесено до `asblockwar.properties`

  в інфо-панелі з'являється посилання **🔍 Пошук шляху до ворожого ресурсу**.
  По кліку виконується зважений алгоритм Дейкстри від поточного вузла до всіх
  досяжних вузлів `PrimaryEnemyResources`. Ваги ребер визначаються типом вузла
  призначення: `ASN` → 1, `AS_SET` → 2, `MNTNER` → 10, `ORGANISATION` → 100.
  Маршрути через AS-простір мають вищий пріоритет над організаційними зв'язками.
  Для кожного знайденого ресурсу показується оптимальний (найменш коштовний)
  шлях із кількістю переходів; результати відсортовано за вагою (найкращий першим).
  Клік по шляху підсвічує всі вузли та ребра на ньому і масштабує вигляд.
  Якщо шляхів немає — виводиться відповідне повідомлення. Граф відмальовується
  як завжди, функція не впливає на розмір файлу помітно (~2 KB JavaScript).

### Виправлено / Покращено

- `GraphExporter.buildJson()` тепер включає масив `primaryEnemyIds` у вбудований JSON
  графа, щоб JS міг ідентифікувати цільові вузли для алгоритму Дейкстри без

## [3.6.15] — 2026-07-18

### Додано

- **Опція `DependencyWithUnknown` (за замовчуванням `false`)**: нова властивість у
  `asblockwar.properties` та checkbox **Include unknown nodes** у діалозі Properties.
  Коли `false` — граф залежностей не містить вузлів зі статусом `UNKNOWN`
  (та ребер до них): прибираються member-ASN без блокуючого статусу, мантейнери і
  організації, на які жодна blocked/suspicious/cleared AS не поширила свій статус.
  Залишаються лише `BLOCKED`, `SUSPICIOUS` і `CLEAR` вузли та їх організаційна
  інфраструктура — саме те, що потрібне для аналізу зв'язаності.
  Коли `true` — поведінка як до цього (повний граф з усіма UNKNOWN).
  Фільтрація виконується в `GraphBuilder.build()` після propagation статусів —
  вузли з вищим статусом зберігаються завжди.

### Покращено

- **Висота вікна Run Progress**: `prefHeight` збільшено з 500 до 700 px — лог
  помітно більший без прокручування.

  `parseAsSetEdges()` тепер також парсить поля `mnt-by:` та `mnt-ref:` з RPSL
  AS-SET-блоків (аналогічно до `parseRpslEdges()` для aut-num). AS-PHIBEE-TELECOM з

- **Ребра з mntner reverse-lookup RPSL (aut-num / as-set)**: новий метод
  `parseMntnerEdges()` парсить RPSL-результат `-rmb` (поля `aut-num:` та `as-set:`)

  додає вузол і ребро. MNT-RU-JSCIOT-1 з `as-set: AS-AOIOT` тепер не є сиротою.

  з'являються в `members:` вже відомих AS-SET-ів, але ще не мають RPSL. Ітерується

  members AS-PHIBEE-TELECOM, тепер мають свої `members:` розпарсені.

- **AS-SET з mntner reverse-lookup додаються до карти до BFS**: AS-SET-и, знайдені

### Додано

- **Завантаження RPSL для ASN-членів AS-SET-ів**: новий метод `fetchMemberAsnRpsl()`

- Головне вікно з акордеоном (вкладки List MNT-BY, List AS-SET, List AS),
  панелями Juniper WAR і blackbgp, кнопками Run та Properties.
- Діалог **Run Progress** з живим логом (GuiLogAppender) та індикатором прогресу.
- Діалог **Properties** для редагування конфігурації.
- Діалог **WhoisInfo**: подвійний клік на елементі списку відкриває RPSL-блок з локальної DB.
- **Live highlighting**: під час обробки активний ASN/AS-SET/MNT-BY підсвічується у відповідній вкладці.
- Кастомний **FilePickerController** (чистий JavaFX, без GTK `FileChooser`).

### Залежності
- Додано `org.openjfx:javafx-fxml:27-ea+6`.

## [1.x] — 2026-04-10 — 2026-07-07

Перший активний цикл розробки після початкового коміту. Версія позначалась
як `1.x` до введення STORE-виходу.

### Ядро обробки (квітень 2026)
- `makeAggressorResources()`: пошук нових ASN з блоків mntby/as-set за `AGGRESSOR_PATTERN`.
- `filterAggressorAsnResources()`: фільтрація мапи за скомпільованим Pattern.
- `storeAggressorAsnResources()`: backup (`list.YYYY-MM-DDThh:mm:ss+HH:mm.txt`) + запис `list.txt`.
- Кеш `ConcurrentHashMap` у `retrieveOrganisation` для уникнення повторних DB-запитів.
- Фінальний звіт у три колонки: **Вилучено | Додано | Модифіковано**.
- Виправлення `AGGRESSOR_PATTERN`: `$` зовні групи, `.*` після ключових слів,
  замінено `String.matches()` на `Pattern.find()`.
- Паралелізм через `FixedThreadPool` (пізніше замінено на virtual threads).
- Логування через Logback (logback.xml з ротацією файлу).
